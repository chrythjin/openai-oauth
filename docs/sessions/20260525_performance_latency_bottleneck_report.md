**작성일**: 2026년 5월 25일  
**대상 버전**: `openai-oauth` monorepo (Bun + Turbo 기반)

## 1. 개요
본 보고서는 `openai-oauth` 프록시 서버 및 핵심 코어 라이브러리의 코드베이스를 분석하여, API 요청 처리 속도를 저하시키거나 스타트업 블로킹 및 불필요한 시스템 리소스를 낭비할 수 있는 성능 병목 요소를 도출하고 이에 대한 구체적인 개선안을 제시하기 위해 작성되었습니다.

분석 결과, 현재 구조에서 **SQLite 데이터베이스 동기식 락(Lock) 경합**, **불필요한 디스크 I/O 반복**, **네트워크 블로킹 체인** 등 성능 향상을 위해 시급히 개선해야 할 치명적인 병목 지점들이 식별되었습니다.

## 2. 병목 현상 상세 분석 및 진단

### 🚨 병목 01 [P0 - 치명적]: SQLite 매 DB 작업마다 Pruning(과거 로그 삭제) 반복 실행
* **관련 파일**: `packages/openai-oauth/src/db.ts`
* **관련 함수/라인**: `insertRequestLog` (라인 221, 269), `getUsageSummary` (라인 273), `getHourlyUsage` (라인 305), `getRecentLogs` (라인 333)
* **상황 진단**:
  * 데이터베이스에 요청 로그를 생성하거나 통계를 조회하는 모든 작업의 시작(및 종료) 부분에서 `pruneOldRequestLogs(db)`가 무조건 호출됩니다.
  * `pruneOldRequestLogs`는 하루 이전의 데이터를 완전히 지우기 위해 `DELETE FROM request_logs WHERE timestamp < ?` 쿼리를 수행합니다.
  * **API 호출 한 번당 총 4번의 `DELETE` 쿼리와 2번의 `INSERT` 쿼리가 동기적으로 실행**됩니다. (요청 로깅 시 전/후 prune 2회 + 응답 로깅 시 전/후 prune 2회)
* **성능 임팩트**:
  * SQLite는 단일 쓰기 락(Write Lock) 모델을 따르기 때문에, 지속적으로 쓰기 및 삭제 쿼리가 경합을 벌이게 됩니다.
  * 데이터가 수천 개 이상 쌓인 상태에서 인덱스가 완전히 타지 않는 대규모 테이블 스캔 형태의 `DELETE`가 매 API 호출 마다 동기 실행되면 프록시 서버 전체의 API 응답 지연 시간이 **수십ms에서 수백ms까지 수직 상승**하며, 결국 연결 타임아웃이나 세션 블로킹으로 이어집니다.
* **개선 방향**:
  * `insertRequestLog` 및 통계 조회 내에서 `pruneOldRequestLogs` 호출을 완전히 제거합니다.
  * **백그라운드 인터벌 타이머**(예: 60초 혹은 5분 주기) 또는 **카운터 기반**(예: 100회 요청 로깅 시 1회만 트리거)으로 비동기 실행되도록 아키텍처를 변경해야 합니다.
  * 데이터베이스 오픈 시 고속 I/O 처리를 위해 **WAL(Write-Ahead Logging) 모드**를 활성화합니다.
    ```typescript
    db.exec("PRAGMA journal_mode=WAL")
    ```

### ⚠️ 병목 02 [P1 - 높음]: 토큰 검증 시 디스크 파일 다중 후보지 검사 반복 (`auth.ts`)
* **관련 파일**: `packages/openai-oauth-core/src/auth.ts`
* **관련 함수/라인**: `readAuthFile` (라인 228), `loadAuthTokens` (라인 321)
* **상황 진단**:
  * `readAuthFile`은 설정된 `authFilePath`가 완전하지 않거나 부재할 때 최대 5개의 디스크 경로 후보지(`resolveAuthFileCandidates`)를 루프 돌며 순차적으로 `fs.readFile`로 조회 및 구문 분석합니다.
  * 비록 `AuthManager`가 메모리에 `this.current`로 토큰을 캐싱하여 평소에는 디스크 접근을 피하지만, 캐시가 만료되거나 토큰 교체(Switch) 또는 관리 명령어 수행 시 후보지를 매번 다시 검사하게 됩니다.
* **성능 임팩트**:
  * 특히 파일 시스템 응답이 느린 가상화 환경이나 Windows HDD, OneDrive 등의 클라우드 동기화 디렉터리 경로 내에 있을 때, 존재하지 않는 다중 디스크 경로를 비동기 루프로 순차 조회하는 시간 자체가 스타트업 및 특정 처리 속도를 떨어뜨립니다.
* **개선 방향**:
  * 최초 구동 시 유효하다고 판정된 한 개의 파일 경로를 캐싱(`activeAuthPath` 등)하여 다음 리로드 시 불필요한 다중 후보지 순회를 원천 차단합니다.
  * 토큰 리로드 간격에 인메모리 최소 TTL(예: 15초~30초) 캐시 레이어를 적용합니다.

### 🐢 병목 03 [P2 - 보통]: Codex 클라이언트 버전 순차적 체인 조회 (`models.ts`)
* **관련 파일**: `packages/openai-oauth/src/models.ts`
* **관련 함수/라인**: `resolveCodexClientVersion` (라인 156)
* **상황 진단**:
  * 서버 구동 단계에서 모델 탐색을 위한 Codex CLI 버전을 가져오기 위해 다음 동작을 동기적으로 순차 수립합니다.
    1. `exec("codex --version")`를 실행하여 로컬 프로세스 확인
    2. 실패 시 외부 npm registry (`https://registry.npmjs.org/@openai/codex`)로 네트워크 요청 전송
    3. 실패 시 내부 하드코딩된 버전을 폴백
* **성능 임팩트**:
  * OS 수준에서 새로운 프로세스(`codex`)를 포크(Fork)하여 실행하는 작업은 수십 ms 이상이 지연되는 무거운 동작입니다.
  * 로컬 CLI가 없어서 실패할 때, 외부 네트워크 상황에 의존하는 npm registry 쿼리가 수행되며, 응답 속도가 느리거나 무응답일 때 **서버가 구동되는 즉시 클라이언트를 서빙하지 못하고 블로킹** 상태에 빠집니다.
* **개선 방향**:
  * 로컬 버전 체크 및 외부 네트워크 요청을 분리하여 **병렬(`Promise.all`)**로 실행하거나, 짧은 타임아웃(예: 300ms)을 지닌 `Promise.race`를 도입합니다.
  * 성공한 버전 문자열을 로컬 디렉터리에 캐싱하여 매번 프로세스 실행 및 네트워크 접근을 원천 배제합니다.

### 📦 병목 04 [P3 - 보통]: SSE 스트리밍 응답 처리를 위한 불필요한 스트림 복제 (`transport.ts`)
* **관련 파일**: `packages/openai-oauth-core/src/transport.ts`
* **관련 함수/라인**: `captureResponsesState` (라인 277 ~ 291)
* **상황 진단**:
  * 프록시 서버의 스트리밍 및 일반 응답 데이터를 가로채기 위해 `response.body.tee()`를 호출해 2개의 스트림으로 쪼갠 뒤 상태 캡처 및 전달을 수행합니다.
* **성능 임팩트**:
  * Web Streams API 표준에서 `.tee()`는 데이터 스트림 청크를 메모리 버퍼에 담아 복제하는 구조이므로 CPU 오버헤드와 함께 가비지 컬렉션(GC) 및 메모리 사용량이 일시적으로 2배 이상 치솟게 됩니다.
  * 특히 대량의 토큰을 고속으로 출력하는 대형 모델 추론이나 긴 텍스트 답변 스트리밍 시, 전체 속도가 미세하게 지연되는 병목이 일어납니다.
* **개선 방향**:
  * 대시보드 상태 저장이 비활성화되어 있거나 이력이 필요 없는 순수 프록시 전송 상황에서는 `tee()`를 아예 거치지 않고 원본 스트림을 바이패스(Bypass)하도록 전처리를 간소화합니다.

### ☁️ 병목 05 [P4 - 낮음]: 구동 시마다 비동기 업데이트 상시 검사 (`update-check.ts`)
* **관련 파일**: `packages/openai-oauth/src/update-check.ts` / `cli-app.ts`
* **상황 진단**:
  * 프록시 서버가 켜질 때마다 NPM 공식 레지스트리에서 `latest` 버전을 조회합니다.
  * 비록 메인 스레드를 블로킹하지 않도록 `void checkForOpenAIOAuthUpdates()` 구조로 흘려보내지만, 격리 네트워크 혹은 인터넷이 매우 느린 오프라인 인프라에서는 계속해서 소켓이 미결 대기 상태로 대기하게 됩니다.
* **개선 방향**:
  * 업데이트 체크 성공 시의 타임스탬프를 DB나 임시 설정 파일에 저장해 두고, 마지막 체크 시간 기준 **24시간 이내에는 NPM 원격 조회를 아예 스킵**하도록 설계합니다.

## 3. 종합 개선 우선순위 권장 로드맵

| 우선순위 | 병목 포인트 | 예상 리스크 | 난이도 | 권장 해결 방안 |
| :---: | :--- | :--- | :---: | :--- |
| **P0** | **매 DB IO마다 Prune DELETE** | 동시 요청 시 전체 프록시 지연 및 DB 잠김 현상 | **하 (Low)** | `insertRequestLog`에서 pruning 호출 제거 후 백그라운드 Interval 타이머 도입 |
| **P1** | **토큰 로드 시 5개 경로 순회** | 디스크 접근 오버헤드 누적 | **중 (Medium)**| 유효 경로 확정 캐시 및 인메모리 TTL 캐시 적용 |
| **P2** | **버전 조회의 순차 블로킹** | 네트워크 문제 시 구동 블로킹 지연 | **중 (Medium)**| 병렬 처리(`Promise.all`) 및 300ms 타임아웃 가드 설치 |
| **P3** | **응답 바디 무조건 `.tee()`** | 스트리밍 성능 저하 및 메모리 오버헤드 | **중 (Medium)**| 로깅/상태 추적 비활성화 시 `tee()` 바이패스 분기 처리 |
| **P4** | **스타트업 상시 업데이트 체크** | 오프라인 인프라 소켓 누수 | **하 (Low)** | 24시간 캐시 기간을 지닌 타임스탬프 방식 제어 |

## 4. 결론 및 제안
`openai-oauth` 시스템은 가볍고 신속한 프록시 전달이 생명인 단일 사용자 localhost 유틸리티입니다. 그러나 현재 적용된 DB 로깅 시스템 및 버전/토큰 감지 시의 동기적, 순차적 처리 설계는 하이 트래픽이나 복잡한 워크플로 실행 시 심각한 프레임 딜레이와 대기 현상을 유발합니다.

제시된 개선안 중 **P0(SQLite Pruning 주기적 스케줄링 및 WAL 모드 도입)** 만 먼저 적용하더라도 프록시 지연 시간을 비약적으로 낮출 수 있습니다. 해당 개선 작업을 순차적으로 적용하여 안정적이고 빠른 로컬 워크스테이션 환경을 구성할 것을 적극 권장합니다.
