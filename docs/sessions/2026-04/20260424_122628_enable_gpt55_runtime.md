# Session Summary - Enable GPT-5.5 in local OpenAI-OAuth runtime

## Goal
Make the local `openai-oauth` repo actually expose and use `gpt-5.5` in the user's OpenCode environment, including the Windows service path.

## What changed
- Updated `packages/openai-oauth/src/models.ts` fallback Codex client version from `0.111.0` to `0.124.0` so model discovery falls back to a version that exposes `gpt-5.5`.
- Kept `packages/openai-oauth/src/types.ts` default model list aligned with current discovered models, including `gpt-5.5` and `gpt-5.4-mini`.
- Changed `register-service.ps1` so the service wrapper no longer hard-pins `--models ...`; it now starts the proxy with `--codex-version 0.124.0`, allowing upstream model discovery to surface `gpt-5.5`.
- Updated docs/examples in `README.md`, `packages/openai-oauth/README.md`, and `packages/openai-oauth-provider/README.md` to show `gpt-5.5` and the new fallback/version guidance.
- Updated tests in `packages/openai-oauth/test/models.test.ts` and `packages/openai-oauth/test/cli.test.ts` for the new fallback version.
- Fixed `packages/openai-oauth/test/live.e2e.test.ts` so live e2e only runs when `LIVE_CODEX_E2E=1`, matching the provider package and keeping default test runs stable.

## Why
The real blocker was not the provider code; it already accepted arbitrary model IDs. The blocker was the local Windows service registration path pinning a stale model allowlist and using an older discovery version path. Removing the hardcoded service allowlist keeps the local runtime aligned with upstream model availability.

## Verification
- `bun run build` ✅
- `bun run test` ✅
- Direct provider call with `gpt-5.5` and `codexVersion: "0.124.0"` returned `gpt-5-5-verified` ✅
- Graphify rebuild completed ✅

## Notes
- The currently running local proxy may still show the old model list until the service is re-registered/restarted with the updated `register-service.ps1`.
- `.serena/cache/...` files changed during inspection but are local cache artifacts and should not be committed.
